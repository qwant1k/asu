## 🔧 БЛОК B — Полнофункциональный модуль заявок

```
ЗАДАЧА: Все 8 типов заявок должны работать от начала до конца — создание, согласование,
подписание через OTP, исполнение, обновление остатков.

═══ BACKEND ═══

1. СОЗДАНИЕ ЗАЯВКИ:
   - Endpoint POST /api/v1/requests/ принимает тип заявки и список позиций
   - Автоматически определяет цепочку согласования по типу заявки:
     * ТМЗ/ОС/НМА → USER → DEPT_HEAD → MOL_WAREHOUSE/MOL_NMA → AHS_HEAD
     * Представительские ТМЗ → только IRD_WORKER/AHS_WORKER могут создавать
     * Перемещение увольняющегося → инициирует DEPT_HEAD или AHS_WORKER
   - Создаёт записи RequestApproval для каждого участника цепочки
   - Отправляет email первому согласующему (через Celery task)

2. OTP-МЕХАНИЗМ (полная реализация):
   a) POST /api/v1/requests/{id}/generate-otp/
      - Генерирует случайный 6-значный код
      - Хэширует SHA-256 с солью (settings.SECRET_KEY)
      - Сохраняет хэш в RequestApproval.otp_code_hash
      - Устанавливает otp_expires_at = now() + 30 минут
      - Отправляет email с кодом (Celery task)
      - Возвращает: {"expires_in_seconds": 1800, "email_sent_to": "***@kfgd.kz"}
   
   b) POST /api/v1/requests/{id}/approve/
      - Принимает: {"otp_code": "123456", "comment": ""}
      - Верифицирует: hash(otp_code) == stored_hash AND not expired
      - При ошибке: {"error": "Неверный код"} или {"error": "Код истёк"}
      - При успехе: обновляет статус заявки, отправляет уведомление следующему
      - Атомарная транзакция

3. ИСПОЛНЕНИЕ ЗАЯВКИ (POST /api/v1/requests/{id}/confirm-receipt/):
   - Только инициатор может подтвердить получение
   - Только в статусе APPROVED
   - Создаёт записи StockMovement (тип: ISSUE)
   - Создаёт/обновляет AssetAssignment для каждой позиции
   - Обновляет WarehouseStock (atomic, с проверкой остатков)
   - Если остаток < запрошенного количества → ошибка с указанием нехватки

4. ПРОВЕРКА ЛИМИТОВ:
   - При создании заявки на ТМЗ: проверить LimitNorm для пользователя/подразделения
   - Если превышает лимит → предупреждение (Warning, не Error) с указанием лимита
   - Предупреждение не блокирует создание, только информирует

5. УВЕДОМЛЕНИЯ (Celery tasks):
   - task: send_request_notification(request_id, event_type, recipient_id)
   - Шаблоны email для событий: CREATED, PENDING_APPROVAL, APPROVED_STEP, 
     FULLY_APPROVED, REJECTED, EXECUTED
   - In-app Notification создаётся синхронно при каждом изменении статуса

═══ FRONTEND ═══

6. Страница /requests/new — выбор вида заявки:
   - 8 карточек с иконками и описанием каждого типа
   - Неактивные типы (нет прав) — задизейблены с подсказкой tooltip
   - Клик → переход на форму конкретного типа

7. Форма заявки (общие требования для всех 8 типов):
   - Header: тип заявки + автоноМер (присваивается после сохранения)
   - Блок «Инициатор»: автозаполнен данными текущего пользователя (readonly)
   - Блок «Позиции»:
     * Select с поиском по наименованию (loadOptions из API)
     * При выборе актива: автозаполнение цены, ед.изм., остатка на складе
     * Поле «Количество» с валидацией: число > 0, не больше остатка
     * Кнопка «+ Добавить позицию», кнопка «Удалить» у каждой строки
     * Итоговая сумма — автоподсчёт
   - Кнопки: «Сохранить черновик» и «Отправить на согласование»

8. Специфика отдельных типов заявок:
   
   a) Заявка на перемещение ОС:
      - Поле «Откуда» (текущий ответственный — Select из AssetAssignment)
      - Поле «Куда» (новый ответственный — Select из Users)
      - Список ОС фильтруется по текущему ответственному
   
   b) Заявка на выдачу ОС новому сотруднику:
      - Поле «Новый сотрудник» — Select с поиском
      - Поле «Дата выхода на работу» — DatePicker
   
   c) Заявка на изменение пользователя НМА:
      - Поле «Текущий пользователь» (автозаполнение из AssetAssignment)
      - Поле «Новый пользователь» — Select
      - Обоснование — Textarea (обязательное)

9. Карточка заявки /requests/{id}:
   - Шапка: номер, тип, дата, инициатор, статус (цветной Badge)
   - Таблица позиций: наименование, кол-во запрошено, кол-во выдано (если исполнена)
   - Timeline согласования: 
     * Каждый этап: роль, ФИО, дата/время действия, статус (иконка)
     * Текущий ожидающий этап — пульсирующая иконка
   - Панель действий (зависит от роли и статуса):
     * Инициатор в DRAFT: кнопки «Редактировать», «Отправить», «Отменить»
     * Согласующий в нужном статусе: кнопки «Согласовать» (→ OTP modal), «Отклонить»
     * Инициатор в APPROVED: кнопка «Подтвердить получение»
   
10. OTP Modal — полная реализация:
    - Заголовок: «Подтверждение подписания»
    - Текст: «Код отправлен на {masked_email}»
    - Input: 6 ячеек (OTP Input компонент Ant Design)
    - Таймер обратного отсчёта: «Код действителен ещё 29:45»
    - Кнопка «Запросить новый код» — активна только после истечения таймера
    - Кнопка «Подписать» — disabled пока код не введён полностью
    - При неверном коде: Input окрашивается в красный + сообщение
    - При успехе: Modal закрывается, статус заявки обновляется без перезагрузки

11. Журнал заявок /requests:
    - Таблица с фильтрами: тип заявки, статус, дата (range), инициатор
    - Разные колонки для разных ролей:
      * Рядовой USER: видит только свои заявки
      * DEPT_HEAD: видит свои + заявки подчинённых
      * MOL/AHS: видят все заявки + колонка «Ожидает моего действия» (badge)
    - Быстрые фильтры (Segmented): «Все» / «Ожидают моего действия» / «Мои»
```
